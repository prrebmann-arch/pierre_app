## 1. ARCHITECTURE ACTUELLE

### 1.1 Stack
- **Frontend**: Vanilla JS (aucun framework), HTML monolithique, CSS custom
- **Backend**: Supabase (auth, DB, storage, RLS), Stripe (payments), Expo Push
- **CDN libs**: Supabase JS v2.49.1, Stripe.js v3, Chart.js v4.4.7, Font Awesome 6.4, Google Fonts (Inter)


### 1.4 Navigation Model
- **SPA via show/hide**: `showSection(name)` hides all `.section` divs, shows target
- **Hash-based persistence**: `location.hash = section` (e.g., `#dashboard`, `#athletes`)
- **Athlete detail**: Sub-navigation via `switchAthleteTab(tab)` — 12 tabs
- **Template tabs**: `switchTemplateTab(tab)` — training, nutrition, workflow, questionnaires
- **Business tabs**: `bizSwitchTab(tab)` — dashboard, leads, automations, content, instagram, messages
- **No router**: All in one page, no URL parameters for IDs

## 7. KEY PATTERNS TO TRANSFORM

### 7.1 DOM Manipulation -> React Components
Every file uses `innerHTML = \`...\`` for rendering. This is the #1 migration effort. All UI construction becomes JSX components with props.

### 7.2 Global State -> State Management
~40 `window.xxx` variables spread across files. Needs:
- **Auth context**: `currentUser`
- **Athletes context/store**: `athletesList`, `currentAthleteId`, `currentAthleteObj`
- **Local component state**: Most `window._xxx` variables are component-local

### 7.3 Event Listeners -> React Handlers
Multiple `document.getElementById('form').addEventListener('submit', ...)` patterns -> `onSubmit` props on forms.

### 7.4 Navigation -> Next.js App Router
```
/                    -> Landing page (public)
/auth                -> Login/Register
/setup-payment       -> Payment wall
/app                 -> Layout with sidebar
/app/dashboard       -> Dashboard
/app/athletes        -> Athletes list
/app/athletes/[id]   -> Athlete detail (tabs via searchParams or nested routes)
/app/bilans          -> Bilans overview
/app/videos          -> Videos section
/app/templates       -> Templates (with tabs)
/app/aliments        -> Aliments database
/app/exercices       -> Exercices database
/app/formations      -> Formations
/app/business        -> Business dashboard (with sub-tabs)
/app/profile         -> Coach profile
/admin               -> Admin dashboard
```

### 7.5 Supabase Queries -> React Query / Server Components
- Read-only data fetching -> Server Components or `useQuery`
- Mutations -> `useMutation` with optimistic updates
- Real-time subscriptions -> Supabase realtime hooks

### 7.6 Chart.js -> react-chartjs-2
4 files use Chart.js directly: apercu.js, bilans.js, business-ig.js. Custom SVG charts in bilans.js need conversion too.

### 7.7 Stripe Integration -> Server Actions
Payment-related code needs to move server-side:
- Setup intents -> API routes
- Webhook handling -> already in `/api/stripe`
- Connect onboarding -> API routes

### 7.8 OAuth Flows -> API Routes
Instagram and Facebook OAuth (business-ig.js, business-messages.js) should use Next.js API routes for token exchange.

## 10. CRITICAL RISKS

1. **Shared mutable state**: `window.xxx` pattern means many modules mutate shared state. Must carefully trace all mutations during migration.
2. **`supabaseClient.auth.signUp` for athletes**: In athletes.js, the coach creates athlete auth accounts client-side, then restores own session. This is fragile and should become a server action.
3. **No TypeScript**: No type safety. Adding TypeScript during migration will slow initial progress but pay off immediately.
4. **innerHTML XSS surface**: Although `escHtml()` is used, the pattern of building HTML strings is error-prone. React's JSX will eliminate this class of bugs.
5. **Chart.js direct DOM**: Charts manipulate canvas directly. Must use react-chartjs-2 wrappers or alternative (Recharts).
6. **50+ Supabase tables**: Need comprehensive type generation (`supabase gen types typescript`).
